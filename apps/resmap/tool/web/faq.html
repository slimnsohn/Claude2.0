ResMap

Prediction-market resolution intelligence — a dataset that knows how markets settle, not just what they cost.

The one idea: anyone can store "market X trades at 42¢." ResMap stores under exactly what conditions X resolves YES, from what authoritative source, by what cutoff, and how that differs from the lookalike market on another venue. That answer is expensive to compile, compounds over time, and is the entire product.

Plain-English glossary

New to prediction markets? Start here — the rest of the page uses these terms.

TermWhat it means
Prediction marketA market where you buy/sell "Yes" or "No" on whether a future event happens. The price (e.g. 42¢) is roughly the implied chance.
VenueA platform that runs these markets. ResMap covers three: Polymarket, Kalshi, Gemini.
Resolution / settleHow a market is decided once the event is known — i.e. whether it pays out "Yes" or "No".
Settlement sourceThe authority that officially decides the outcome (e.g. "FIFA", "NASA", "the Associated Press").
CutoffThe exact deadline/time that counts for settlement (a 6pm vs 11:59pm cutoff can capture different facts).
Arbitrage ("arb")Buying the same outcome cheap on one venue and selling it dear on another for a "risk-free" profit — if the two markets truly resolve the same way.
false friendTwo markets that look identical but can resolve differently — so the "free" arb is actually a trap. Surfacing these is ResMap's whole point.

The big picture

What is ResMap, in one paragraph?

ResMap continuously pulls every market from Polymarket, Kalshi, and Gemini, stores each market's settlement rules text verbatim, uses Claude to extract the rules into structured fields (a human reviews every extraction), and then compares lookalike markets across venues to determine whether they would actually resolve identically. The output is a Postgres dataset you can query: markets, immutable rule history, structured settlement semantics, and a cross-venue equivalence graph with risk scores.

Why is this valuable? Isn't this just an arb scanner?

No — that's the point. A naive arb scanner sees "France wins World Cup" at 20¢ on one venue and 24¢ on another and calls it free money. But two markets on the same real-world event can resolve differently: different settlement sources can call the event differently, a 6pm vs 11:59pm cutoff captures different facts, draws may void one market and resolve the other NO, and ">50%" vs "≥50.0%" flip on the boundary. ResMap's edge is knowing which lookalike pairs are false friends — the traps. Price arithmetic is a commodity; settlement semantics are not.

What are the three data layers?
  1. Market registry (markets) — every market per venue: ids, titles, timestamps, status, outcome. Table stakes; competitors sell this too.
  2. Resolution-rule extraction (rule_snapshots → parsed_rules) — verbatim rules text captured immutably, then LLM-parsed into structured fields and human-reviewed. This is where the value starts.
  3. Cross-venue equivalence (equivalences) — for markets that look identical: do they actually resolve identically? The crown jewel; nobody else has it cleanly.

The sources and equivalences tables are the proprietary IP.

Layer 1 — Ingestion

What does a venue adapter do?

Each adapter (ingest/adapters/polymarket.py, kalshi.py, gemini.py) fetches its venue's public API and yields a common MarketRecord: venue id, title, the verbatim settlement rules text, timestamps, status, and the full raw API blob for forensics. Adapters are deliberately pure — fetch and yield, no database writes — so all storage logic lives in one place (ingest/core.py) and a venue can be added without touching the core. None of the three venues requires credentials for market data.

  • Polymarket: Gamma API; the market description field is the settlement text (HTML is stripped, structure preserved).
  • Kalshi: rules_primary + rules_secondary; Kalshi names its settlement source most explicitly of the three.
  • Gemini: an event holds multiple contracts (the tradable YES/NO instruments); each contract's rich-text description holds the rules and becomes its own market row.
How does change detection work — and why does it matter?

ingest/core.py normalizes each market's rules text (whitespace, casing) and hashes it (SHA-256). On every ingest run, the hash is compared to the latest stored snapshot:

  • Same hash → nothing written. Re-running ingest a hundred times adds zero noise.
  • New market → registry row + first snapshot.
  • Changed hash → a new immutable snapshot is appended, a rule_change_events row is written, and every existing parse of that market is flagged stale for re-parsing.

"The venue edited the settlement criteria mid-market" is itself high-value risk information — traders holding positions want to know immediately. That alert feed falls out of the hashing for free.

Why is the raw rules text append-only?

Two reasons. Auditability: every structured field in parsed_rules references the exact snapshot it was derived from, so any claim in the dataset can be traced to "this is the text we read, fetched at this time." History is signal: overwriting rule text would destroy the record of what changed mid-market. Nothing in ResMap ever updates or deletes a snapshot.

Layer 2 — Rule parsing

What does the LLM extract from the rules text?

parse/rule_parser.py sends each snapshot's verbatim text to Claude (via the local claude -p CLI — no API cost) with a fixed contract: return exactly nine fields, JSON only, think "like a lawyer reading a contract, not a casual bettor reading a headline."

FieldWhat it captures
resolution_logicnormalized "resolves YES if …"
authoritative_sourcethe primary authority as a short canonical entity — "FIFA", "AP", "NASA GISS land-ocean index" (so the same authority dedupes across markets)
source_fallbackthe secondary procedure if the primary is silent — e.g. "consensus of credible reporting"
source_typeofficial_data / media_call / exchange_discretion / onchain / other
cutoff_time + cutoff_basisthe actual settlement cutoff and how it's defined
tie_handlingwhat happens on a tie / draw / push
revision_handlingwhat happens if the source data is later revised
threshold_defexact threshold and rounding — "≥ 50.0%" vs "> 50%"
confidencethe model's own 0–1 self-assessment

If the text doesn't specify a field, it is stored as null — missing information is shown as missing, never invented. Markets are parsed highest-volume first; there's no cutoff, lower-volume markets just wait for a later batch.

Why does a human review every parse?

Because the review is the moat. LLM extraction is cheap and cloneable; a curated, human-verified settlement-semantics database is not. Every parse lands with reviewed = false. The review CLI (parse/review_cli.py) lists unreviewed parses lowest-confidence first, shows each parsed interpretation side-by-side with the verbatim snapshot text it came from, and only a human can flip a parse to approved. There is deliberately no auto-accept path anywhere in the codebase.

python -m parse.review_cli list
python -m parse.review_cli show 7d45baf9
python -m parse.review_cli approve 7d45baf9
What is the sources table?

A normalized registry of authoritative settlement sources. When the parser extracts "AP race call" from twelve different markets, they all point at one sources row. That lets you ask questions no competitor can answer without doing this curation work — "show me every market across all venues that settles off the BLS CPI release." It also powers the most important equivalence axis: two markets sharing a source row are verifiably settled by the same authority.

What happens when a venue edits rules after parsing?

The next ingest run detects the new hash, appends the new snapshot, and flags all of that market's parses is_stale = true. The parser's selection query automatically re-picks stale markets on the next run and parses the new text; old parses are kept as history. No special re-parse code path exists — staleness simply re-enters the normal queue.

Layer 3 — Equivalence (the moat)

How are lookalike pairs found?

parse/candidate_matcher.py does a cheap, high-recall sweep: fuzzy title similarity (rapidfuzz token-sort, threshold 0.65) across open Polymarket ↔ Kalshi markets, blocked by close-date proximity. It deliberately over-matches — missing a real pair is the costly error — and persists nothing. Pairs flow straight into the equivalence engine, which does the careful work.

What are the four divergence axes?
AxisWeightWhy it can flip resolution
source0.45a different authoritative source can call the same event differently
cutoff0.25a 6pm vs 11:59pm cutoff captures different facts
threshold0.20"≥50.0%" vs ">50%" and rounding rules differ on the boundary
tie0.10draw/push/void handling differs

For each candidate pair, the engine compares the two parsed interpretations axis by axis. Wherever it can decide deterministically, it does (shared sources row → same source; equal timestamps → same cutoff; identical normalized text → same). Only genuinely ambiguous axes — paraphrased tie clauses, one-sided nulls — go to a Claude judgment call, prompted to flag a difference only when it could plausibly change which way money is paid.

What do the match types mean?

The weights of all differing axes are summed into a risk_score (0 = will resolve identically, 1 = will plausibly resolve differently):

Match typeRiskMeaning
true_match0identical on all four axes — safe to treat as the same market
near_match0 < r ≤ 0.25differ on a minor axis — flag, don't trust blindly
false_friend> 0.25differ in a way that can flip resolution — the trap a naive arb scanner calls free money

Surfacing false_friends is the product's edge over every cheap arb tool. Results are upserted into equivalences with the axes that diverged and human-readable notes; re-runs update in place.

Operations

What's the day-to-day workflow?
# 1. pull latest markets + detect rule changes (all 3 venues)
python -m ingest.run

# 2. parse the highest-volume unparsed markets, e.g. 20 at a time
python -m parse.rule_parser --limit 20

# 3. review the new parses (shakiest first)
python -m parse.review_cli list

# 4. find lookalike pairs and score divergence
python -m parse.equivalence

Each step is idempotent — running it twice never duplicates data. Maintenance is bounded: keep three adapters alive, review parses, re-review markets whose rules changed.

What does ResMap deliberately NOT do?
  • No trading. Read-only by design — no custody, no order execution, no wallets. ResMap sells the signal; the buyer executes. This keeps the legal and maintenance surface near zero.
  • No auto-accepted parses. Human review is mandatory.
  • No invented data. Unparseable fields are null; markets with no rules text (e.g. Kalshi multi-leg parlays) stay registry-only.
  • No equivalence beyond Poly↔Kalshi yet. Gemini is ingested but stays out of the equivalence graph until the Poly↔Kalshi layer is solid.

Where does the money come from?

Three surfaces, all riding the same dataset:

  1. Metered read API over markets / parsed rules / equivalences — for tool-builders (B2B).
  2. Alert feeds — rule-change events and new false-friend detections as a subscription.
  3. Parquet historical export — for backtesters and researchers.

The arb/alert web tool (Phase 4) is a demo that sells #1–#3, not the product itself.
